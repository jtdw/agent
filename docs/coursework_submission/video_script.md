# 视频讲解稿

## 片头

大家好，我这次课程作业完成的是一个 GIS 智能体系统。它不是一个普通聊天机器人，而是一个面向地理数据处理的智能工作台。

它的核心目标是：用户上传地理数据后，可以直接用中文描述任务，系统自动识别数据类型，选择合适的 GIS 工具或工作流，完成空间分析、地图预览、结果解释和产物下载。

## 项目整体界面

打开系统后，可以看到它由几个主要区域组成：文件上传区、智能体对话区、地图预览区、任务结果区和产物下载区。

用户不需要一开始就知道具体调用哪个 GIS 函数，只需要上传数据，然后用自然语言说明需求。例如：“帮我查看这个矢量数据的范围和坐标系”，或者“把这个表格中的经纬度转换为点图层并显示在地图上”。

## 典型使用流程

第一步，用户上传地理数据。系统支持常见的矢量、栅格和表格数据，例如 Shapefile、GeoJSON、TIFF 和 CSV。

第二步，后端会识别数据类型、字段、坐标系和空间范围，并把这些信息绑定到当前用户和当前会话中。

第三步，用户在对话框中输入中文任务。智能体不会完全依赖大模型自由发挥，而是结合工具卡片、运行时上下文和稳定工作流来判断应该执行什么操作。

第四步，系统执行 GIS 工作流。例如矢量裁剪、栅格统计、表格经纬度转点、地图制图，或者土壤湿度 XGBoost 建模和 GeoConformal Prediction 不确定性分析。

第五步，结果会注册为 artifact，并在地图和结果面板中展示。用户可以查看结果摘要，也可以下载生成的文件。

## 系统结构说明

从结构上看，前端使用 React 和 TypeScript 构建 GIS 工作台界面，负责上传、对话、地图预览和结果展示。

后端使用 FastAPI，负责用户请求、文件管理、会话绑定、工具执行和结果返回。

中间的 Agent Runtime 是智能体运行时，它负责把用户输入、上传文件、工具卡片和运行上下文组织起来，再选择合适的 GIS 工具或稳定工作流。

GIS 工具链负责真正的数据处理，包括矢量、栅格、表格、制图、建模和 artifact 管理。

系统还保留了 staging gate、CI 测试和 rollback 机制，用来降低运行时切换和功能扩展带来的风险。

## 功能亮点

第一个亮点是 GIS 工作流稳定化。系统不是让大模型随意生成代码，而是把高频 GIS 任务做成可复现的工具和工作流。

第二个亮点是中文兼容。项目中特别处理了 UTF-8、中文文件名、中文字段和中文提示词，避免 Windows 默认编码导致乱码。

第三个亮点是结果可追踪。每次工具执行都会生成结构化结果，并注册为 artifact，方便前端展示和下载。

第四个亮点是土壤湿度建模能力。系统已经加入了本地 ISMN 归档思路、XGBoost 预测和 GCP 不确定性输出，适合进一步扩展到地理环境分析任务。

第五个亮点是工程安全边界。项目有 CI、smoke test、staging observation gate 和 rollback 方案，不会因为本地测试通过就直接扩大真实流量。

## 当前交付状态

目前项目已经完成本地功能、核心 runtime、staging gate、soil moisture/GCP gate 和 CI 层面的收尾。代码已经合并到 main，最新 CI 通过，工作区也保持干净。

真实远端 staging 还没有执行，因为还缺少真实 staging 地址、admin token 获取方式、服务重启方式和部署配置核验路径。因此我在项目文档中明确把它标为后续运营接入项，而不是夸大为已经部署到真实远端环境。

## 结尾

总结来说，这个项目实现了一个面向 GIS 数据处理的智能体工作台。它把自然语言交互、GIS 工具链、地图展示、结果下载和工程化验证结合在一起，让用户可以更低门槛地完成常见地理数据分析任务。

以上就是我的课程作业项目展示。
